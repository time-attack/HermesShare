# HermesShare ‚Äî Project Brief

## What this is
A real, native iOS **iMessage App Extension** (not a web view, not App Clip) that renders
**dynamic, data-driven mini-app-like cards** inside iMessage ‚Äî tracking status, small
interactive widgets, progress views ‚Äî where the content and layout are generated by Hermes
(an AI agent) per-message as JSON, not as compiled Swift. This mirrors how Scriptable and
Widgy work: one fixed, native SwiftUI renderer ships once via Xcode; an external JSON payload
describes which of a fixed vocabulary of widgets to show and how to lay them out. No new
code is ever compiled or executed on-device ‚Äî this is a strict, real Apple platform
constraint (no dynamic code execution/download is allowed by the App Store), and the whole
point of the JSON-driven renderer is to deliver the *feeling* of "the agent writes custom UI
per message" without violating that constraint.

## Why (context)
The user runs "Hermes," an AI agent system connected to iMessage via a bridge called Photon
(uses the `spectrum-ts` SDK, project already has Photon set up ‚Äî see
`~/.hermes/skills/software-development/photon-interactive-content/SKILL.md` for how Hermes
currently sends web-based mini-app cards over Photon). Photon's existing `app(url)` mechanism
opens a URL in Photon's own pre-approved webview and needs no Apple Developer account ‚Äî
that's the *other* way to send rich content, already working today. HermesShare is a NEW,
separate, more ambitious mechanism: a REAL native iMessage Extension with actual
`MSMessageTemplateLayout` / native SwiftUI UI, which requires this Xcode project, an Apple
Developer account (user has a local development signing identity already: "Apple Development:
Sina Matian (TA89948CWL)" ‚Äî sufficient for simulator/device testing, not yet for App Store
distribution), and produces a genuinely native-feeling in-thread mini-app experience instead
of a webview.

## Architecture

1. **`Shared/` Swift package/target** ‚Äî the data model + renderer, used by both the host app
   and the extension:
   - `HermesLayout.swift` (already drafted at `Shared/Sources/HermesLayout.swift` ‚Äî READ IT
     FIRST, it's the starting contract for the JSON schema: a `HermesLayout` root document
     containing a `HermesNode` tree ‚Äî vstack/hstack/text/icon/statusBadge/progressRing/
     progressBar/divider/spacer/keyValueRow/mapPreview/image/card). You need to:
     - Write full `Codable` conformance for the `indirect enum HermesNode` (associated-value
       enums need custom `init(from:)`/`encode(to:)` ‚Äî a `type` string discriminator field
       plus per-case payload keys is the standard pattern; put this in a new
       `HermesLayoutCodable.swift` file, keep it separate from the schema definition file for
       readability).
     - Extend/adjust the schema as needed while building the renderer if you find gaps, but
       keep the "fixed vocabulary of native SwiftUI primitives, described by JSON" architecture
       intact ‚Äî do not add any mechanism that evals/interprets arbitrary code strings.
   - `HermesLayoutRenderer.swift` ‚Äî a SwiftUI `View` that takes a `HermesLayout` and renders it
     recursively via a `HermesNodeView` that switches over `HermesNode` cases. This is the
     actual "renderer" ‚Äî analogous to Scriptable's `ListWidget`/`WidgetText` native primitives
     interpreting a script's declarative calls, or Widgy's JSON-driven layout engine.
   - Include a handful of realistic **sample `HermesLayout` JSON fixtures** (as `.json` test
     files or inline Swift literals) covering: (a) a package-tracking status card (progress
     bar + status badge + key-value rows for carrier/ETA), (b) a simple stat/dashboard card
     (2-3 stacked keyValueRows + a progressRing), (c) a map-preview card with a location pin
     + label. Use these to drive both Xcode Previews and a debug harness.

2. **`HermesShare/` ‚Äî the host app target.** A minimal SwiftUI iOS app whose main job is (a)
   satisfying Apple's requirement that an iMessage Extension have a container app, and (b) a
   simple debug/preview screen that lets you paste or pick a sample `HermesLayout` JSON and
   see it rendered full-screen via the Shared renderer ‚Äî this is your fastest iteration loop
   for testing the renderer without going through the full iMessage extension flow every time.

3. **`HermesShareExtension/` ‚Äî the actual iMessage App Extension target** (`MSMessagesAppViewController`
   subclass + `MessagesViewController.swift`). This is the real native extension: when a user
   receives a HermesShare message in iMessage, tapping/viewing it should show the
   `HermesLayoutRenderer` rendering whatever `HermesLayout` JSON was embedded in/referenced by
   the message. For a first working version, the JSON can be embedded directly in the
   message's `MSMessage.url` query string (base64-encoded compact JSON) or `summaryText` +
   a companion `MSSession` ‚Äî pick whichever is simpler to get working first; document your
   choice. A `compactPresentationStyle` (small bubble in the transcript) and
   `expandedPresentationStyle` (full mini-app view when tapped) should both render via the
   same `HermesLayoutRenderer`, at different sizes.

4. **Round-tripping actions back to Hermes.** `HermesAction.deepLinkURL` fields in the JSON
   schema are meant to let a card have tappable actions (e.g. "Mark as delivered", "View full
   tracking"). For a first version, wire tapping an action button to open the deep-link URL
   via `UIApplication.shared.open(url)` (or `MSMessagesAppViewController`'s equivalent) ‚Äî
   actually delivering that back to the Hermes/Photon backend (a webhook endpoint) is a
   stretch goal, not required for v1; a TODO comment and a stub function is fine if you run
   out of scope, just make sure the button UI itself works and is documented as not-yet-wired.

## Tooling / setup

- Use `xcodegen` (already installed at `/opt/homebrew/bin/xcodegen`) with a `project.yml`
  defining 3 targets: `HermesShare` (iOS app), `HermesShareExtension` (iMessage extension,
  embed in HermesShare), and ideally a `Shared` framework/local Swift package the other two
  both depend on (a local SPM package under `Shared/` with a `Package.swift` is cleanest ‚Äî
  xcodegen supports `packages:` + `dependencies: [{package: Shared}]` per target).
- Deployment target: iOS 17.0 minimum (this is a real extension meant to actually ship/test
  on-device eventually; don't require iOS 26 unless you specifically need a feature from it).
- Bundle ID prefix: `com.hermesshare.app` (extension: `com.hermesshare.app.MessagesExtension`
  ‚Äî Apple requires the extension's bundle ID to be `<container>.{something}`, verify the
  exact Apple convention as you scaffold ‚Äî I believe it must literally be a suffix of the
  container app's bundle ID).
- Code signing: `CODE_SIGN_STYLE: Automatic`, `DEVELOPMENT_TEAM` should resolve to the
  existing local identity ("Apple Development: Sina Matian (TA89948CWL)" ‚Äî team ID prefix
  visible in `security find-identity -v -p codesigning` output, dig out the actual team ID
  string, likely `TA89948CWL` itself or discoverable via
  `security find-identity -v -p codesigning` / Xcode's own signing UI logic) so it builds
  without prompting for a paid account.
- Build and run in the **iOS Simulator** (Messages app extensions ARE testable in Simulator's
  Messages app since Xcode lets you pick the "Messages" scheme target and it launches
  Simulator's Messages.app with your extension available to insert) ‚Äî verify this actually
  works end-to-end: `xcodebuild ... -scheme HermesShareExtension ...` or via the host app
  scheme with the extension embedded, then manually confirm (screenshot via
  `xcrun simctl io <udid> screenshot`) that the extension actually appears and renders a
  test payload inside Simulator's Messages app.

## Constraints / do NOT do

- Do not attempt to make the app "compile and run arbitrary Swift code sent by the agent" ‚Äî
  that's not possible on iOS and isn't the actual architecture; the JSON-driven renderer IS
  the correct mechanism, don't second-guess this and try to build a Swift interpreter/JIT.
- Do not integrate with Photon/spectrum-ts in this session ‚Äî this is a standalone iOS Xcode
  project. The Hermes-side "generate a HermesLayout JSON and send it" integration is separate,
  future work.
- Don't publish anything to the App Store or attempt real distribution provisioning ‚Äî local
  development signing + Simulator testing is the actual scope here.
