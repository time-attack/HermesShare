# HermesShare

A **real, native iOS iMessage App Extension** that renders **dynamic, data-driven cards**
inside iMessage — package-tracking status, small dashboards, map previews — where the content
and layout are described per-message as **JSON** generated by Hermes (an AI agent), not as
compiled Swift.

One fixed, native SwiftUI renderer ships once via Xcode. An external JSON payload
(`HermesLayout`) selects and parameterizes a **fixed vocabulary** of native SwiftUI primitives.
No new code is ever compiled or executed on-device — this is the same architecture as
Scriptable's `ListWidget`/`WidgetText` or Widgy's layout engine, and it is the correct (and
only App-Store-legal) way to get the feeling of "the agent writes custom UI per message"
without dynamic code execution.

> **This repo is the standalone iOS Xcode project only.** Wiring it to the actual Hermes /
> Photon send pipeline is intentionally out of scope (see [NEXT STEPS](#next-steps)).

## Architecture

```
HermesShare/                 ← this Xcode project
├─ Shared/                   ← local Swift package (linked statically into both targets)
│  └─ Sources/HermesShared/
│     ├─ HermesLayout.swift          the schema: HermesLayout → indirect enum HermesNode tree
│     ├─ HermesLayoutCodable.swift   hand-written Codable (type-discriminator wire format)
│     ├─ HermesLayoutRenderer.swift  the SwiftUI renderer (the "interpreter")
│     └─ HermesSampleLayouts.swift   realistic fixtures (package / dashboard / map)
├─ HermesShare/              ← host / container app (SwiftUI). Debug harness for the renderer.
└─ HermesShareExtension/     ← the real iMessage extension (MSMessagesAppViewController)
```

### The schema (`HermesLayout`)

A `HermesLayout` is a root document (`version`, `title`, `subtitle`, `accentColorHex`,
`background`, `root`, `actions`) whose `root` is a recursive `HermesNode` tree. The fixed
node vocabulary:

`vstack` · `hstack` · `text` · `icon` · `statusBadge` · `progressRing` · `progressBar` ·
`divider` · `spacer` · `keyValueRow` · `mapPreview` · `image` · `card`

Each node encodes as a JSON object with a `"type"` string discriminator plus that case's
payload keys, e.g.:

```json
{
  "version": 1,
  "title": "Package Out for Delivery",
  "subtitle": "Order #HS-48213",
  "accentColorHex": "#34C759",
  "root": {
    "type": "vstack", "spacing": 12,
    "children": [
      { "type": "statusBadge", "label": "Out for delivery", "colorHex": "#34C759" },
      { "type": "progressBar", "value": 0.78, "colorHex": "#34C759" },
      { "type": "keyValueRow", "key": "Carrier", "value": "UPS Ground" }
    ]
  },
  "actions": [
    { "id": "track", "label": "View full tracking", "deepLinkURL": "hermesshare://action?id=track" }
  ]
}
```

`role`/`weight` on text styles are optional (lenient decoding fills defaults), so Hermes can
emit compact documents. Unknown node `type`s throw rather than silently no-op.

### The renderer (`HermesLayoutRenderer`)

A SwiftUI `View` that walks the tree via a recursive `HermesNodeView` which `switch`es over
every `HermesNode` case and maps it to a native primitive (VStack/HStack, `Text` with system
font roles, SF Symbols, a trimmed-`Circle` progress ring, `ProgressView`, a non-interactive
MapKit `Map` preview, `AsyncImage`, material-backed cards, etc.). The same renderer drives
both the small in-transcript bubble (`.compact`) and the full tapped-open view (`.expanded`);
only chrome/padding differ.

## JSON payload delivery mechanism (current)

**Embedded-in-`MSMessage.url`.** When a card is inserted, its `HermesLayout` is encoded to
compact JSON, then **base64url**-encoded and placed in a query item of the message URL:

```
hermesshare://card?p=<base64url-compact-json>
```

The receiving side reads `conversation.selectedMessage?.url`, pulls the `p` query item, and
decodes it back into a `HermesLayout` (`HermesLayout.decode(base64URLPayload:)`). The
in-transcript bubble also carries a native `MSMessageTemplateLayout` (caption / subcaption /
thumbnail rendered from the card) so it looks native even before it's tapped.

**Why this over the alternatives:** it needs no server, no `MSSession` state juggling, and no
attachment files — the entire card travels inside the message itself, which is the simplest
thing that fully works for v1. The tradeoff is a practical payload-size ceiling (URLs aren't
meant to be huge); a future version can switch to an attachment or a short-code that the
extension resolves against a Hermes endpoint. See [NEXT STEPS](#next-steps).

## Build & run

Requirements: Xcode 26+, an iOS 17+ Simulator, `xcodegen` (`brew install xcodegen`), and a
local Apple Development signing identity (this repo is wired to team `6PPS68Y9RP` — change
`DEVELOPMENT_TEAM` in `project.yml` to yours).

```bash
# 1. Generate the Xcode project from project.yml
xcodegen generate

# 2. Build the host app + embedded extension for a booted simulator
xcodebuild -project HermesShare.xcodeproj -scheme HermesShare \
  -destination 'platform=iOS Simulator,id=<UDID>' -derivedDataPath build/DD build

# 3. Run the unit tests (schema Codable round-trip, base64url transport, lenient decoding)
xcodebuild -project HermesShare.xcodeproj -scheme HermesShare \
  -destination 'platform=iOS Simulator,id=<UDID>' -derivedDataPath build/DD test
```

### Try the renderer (host app)

Launch the **HermesShare** app in the Simulator. The debug harness lets you flip between the
three sample cards, and the `{}` toolbar button drops into a live JSON editor — paste any
`HermesLayout` and watch it render (with inline validation errors). This is the fastest
iteration loop for the renderer.

### Try the extension (Messages.app)

1. Install the host app to the Simulator (it embeds the extension).
2. Open **Messages** in the Simulator, open any conversation, tap the Apps (`⌵`) row, and
   pick **HermesShare**.
3. The compose gallery shows the sample cards — tap one to insert it into the conversation.
   The inserted bubble renders via the shared renderer; tapping it expands to the full card.

## Round-tripping actions back to Hermes

`HermesAction.deepLinkURL` fields let a card carry tappable actions (e.g. "Mark as delivered",
"View full tracking"). Tapping a button opens its `hermesshare://action?...` URL:

- In the **host app**, `onOpenURL` catches it and shows a confirmation sheet.
- In the **extension**, it's opened by walking the responder chain to the shared application.

**Not yet wired:** actually delivering the tap to a Hermes / Photon **webhook** is a stub
(`TODO(hermes-roundtrip)` in `MessagesViewController` / `HermesActionHandler`). The button UI
works today; the network round-trip is future work.

## NEXT STEPS

These connect this standalone project to the live Hermes / Photon pipeline (all out of scope
for this session):

1. **Hermes-side layout generation.** A Hermes skill that emits `HermesLayout` JSON (this
   repo's `HermesSampleLayouts` are the reference shapes). Validate against the schema before
   sending.
2. **Send path via Photon / `spectrum-ts`.** Photon currently sends web mini-app cards via its
   `app(url)` webview mechanism. HermesShare needs a *native-message* send path — inserting an
   `MSMessage` (with the base64url payload) into a thread, which requires the extension to be
   installed on the sending device. Decide whether Hermes drives an on-device shortcut or a
   Messages-integration.
3. **Action webhook.** Replace the `TODO(hermes-roundtrip)` stub with a real POST to a Hermes
   endpoint so `deepLinkURL` taps (e.g. "Mark as delivered") reach the agent. Include the
   message/session identity so Hermes can update the same card.
4. **Payload scaling.** If cards outgrow a URL, switch transport to an `MSMessage` attachment
   or a short id the extension resolves against a Hermes endpoint (keep the URL form as the
   zero-dependency fallback).
5. **Distribution.** This session is local development signing + Simulator only. Shipping to
   real devices / TestFlight needs a paid Apple Developer account and provisioning for both the
   app and the extension bundle IDs (`com.hermesshare.app` and
   `com.hermesshare.app.MessagesExtension`).

## Constraints (by design)

- **No dynamic code execution.** The JSON only *selects and parameterizes* a fixed native
  vocabulary. There is no Swift interpreter/JIT and there will never be one — that's both
  impossible on iOS and against App Store rules. The renderer *is* the mechanism.
- **No Photon / `spectrum-ts` integration here.** This is the standalone iOS project.
- **No App Store distribution.** Local development signing + Simulator testing only.
